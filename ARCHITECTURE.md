# Architecture

## Overview

PERK Sandbox is a compile-time code generation framework. The `#[sandbox_program]` proc macro reads your `sandbox.toml` configuration and generates a Solana program entrypoint that wraps your Anchor business logic with a 12-step safety pipeline. The generated code is the actual program entrypoint — there is no bypass path.

## System Design

```
┌─────────────────────────────────────────────────────┐
│                  Solana Runtime                       │
│                                                       │
│  ┌─────────────────────────────────────────────────┐ │
│  │         sandbox_process_instruction()            │ │
│  │         (generated by #[sandbox_program])        │ │
│  │                                                   │ │
│  │  [1]  Reentrancy guard ─── set executing + depth │ │
│  │  [2]  Mode checks ──────── paused / close-only   │ │
│  │  [3]  Unknown IX check ── fail-closed             │ │
│  │  [4]  Pre-checks ────────  rate limits, oracle    │ │
│  │  [5]  TX snapshot ──────── fingerprint + anchor   │ │
│  │  [6]  IX snapshot ──────── before-values          │ │
│  │                                                   │ │
│  │  [7]  ┌──── Per-Instruction Wrapper ────────┐    │ │
│  │       │  Authority checks                    │    │ │
│  │       │  Bound pre-checks                    │    │ │
│  │       │  Before snapshots                    │    │ │
│  │       │  YOUR BUSINESS LOGIC                 │    │ │
│  │       │  After snapshots                     │    │ │
│  │       │  Invariant post-checks (18 types)    │    │ │
│  │       │  TX cumulative decrease              │    │ │
│  │       └──────────────────────────────────────┘    │ │
│  │                                                   │ │
│  │  [8]  Post-checks ──────── (delegated to wrapper) │ │
│  │  [9]  Circuit breakers ── TVL cliff, events, ptx  │ │
│  │  [10] PDA write-back ──── mode flags, counters    │ │
│  │  [11] Guard cleanup ───── decrement + clear       │ │
│  │  [12] Return OK                                   │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

## Two Crates

### `sandbox-macros` (proc macro)

Runs at compile time. Reads `sandbox.toml`, parses `#[invariant]`, `#[authority]`, `#[bound]` attributes from your instruction functions, and generates:

- The 12-step entrypoint function (`sandbox_process_instruction`)
- Per-instruction wrapper functions with pre/post checks
- Dispatch arms mapping Anchor sighashes to wrappers
- Circuit breaker category routing
- Mode check instruction allowlists

Key modules:
- `program_attr.rs` — Main entrypoint codegen (steps 1-12)
- `lib.rs` — Per-instruction wrapper generation, config wiring
- `invariant_attr.rs` — 18 invariant type codegen with CU reservation
- `snapshot.rs` — Before/after field read codegen (2 strategies)
- `tx_anchor_codegen.rs` — TX fingerprint + cumulative decrease
- `config.rs` — Full `sandbox.toml` parser with typed structs

### `sandbox-runtime` (on-chain library)

Runs at runtime on Solana validators. Provides the actual safety check functions called by the generated code:

- `circuit_breaker.rs` — TVL cliff (HWM ring buffer), event counters, per-IX threshold
- `guard.rs` — Reentrancy detection and depth tracking
- `oracle.rs` — Freshness and deviation validation
- `rate_limit.rs` — Window-based counter management
- `state.rs` — PDA byte-level serialization with bounds-checked access
- `tx_anchor.rs` — Transaction fingerprinting and anchor snapshots
- `cu.rs` — Compute unit budget assertions
- `error.rs` — 25 error codes starting at 6000

## PDA Layout

The sandbox maintains its own PDA (`seeds = [b"perk_sandbox", &[bump]]`) with a byte-exact layout:

```
Header (64 bytes):
  [0..8]    Discriminator: "PRKSANDX"
  [8]       Schema version (currently 1)
  [9]       PDA bump
  [10..12]  Total size (u16)
  [12..14]  Mode flags section offset (u16)
  [14..16]  Circuit breaker section offset (u16)
  [16..18]  Rate limits section offset (u16)
  [18..20]  Guard section offset (u16)
  [20..22]  Emergency offset (u16, alias for mode flags)
  [22..24]  TX anchor section offset (u16)
  [24..26]  Event counters section offset (u16)
  [26..64]  Reserved

Sections (variable-length, at header-specified offsets):
  Mode Flags:     60 bytes (4 bools + slots + reason + cooldown + bypass)
  Guard:          2 bytes (executing + depth)
  TX Anchor:      33 + N×42 bytes (fingerprint + field count + fields)
  Circuit Breaker: 1 + N×(37 + history_len×24) bytes (categories + ring buffers)
  Rate Limits:    1 + N×13 bytes (counter count + counters)
  Event Counters: 1 + N×21 bytes (counter count + id + data per counter)
```

All reads and writes are bounds-checked. Offset validation runs at deserialization. Corruption → `PDACorrupted` error → fail-closed.

## Circuit Breaker Design

Three independent breaker types:

**TVL Cliff** — Windowed high-water mark tracking. Stores recent TVL snapshots in a ring buffer. Compares current value against the window maximum. Per-category budgets (e.g., withdrawals max 15%, liquidations max 25%). The R3-1 fix uses `before_value` as a floor when HWM decays to zero after market inactivity.

**Event Counters** — Track rapid event frequency within a window. Example: "if more than 50 liquidations happen in 1000 slots, pause the program." Window resets cleanly on expiry.

**Per-Instruction Threshold** — Stateless single-instruction check. Catches large single-tx drains. Complementary to TVL cliff which catches cumulative drain via HWM.

Deferred actions: breakers can fire `Pause`, `CloseOnly`, or `PauseLiquidations` as deferred mode flag changes (current tx succeeds, next tx is restricted) or `RejectCurrent` (current tx reverts).

## Invariant System

18 built-in invariant types, all preceded by CU reservation:

| Type | Description |
|------|-------------|
| `gte` / `lte` / `eq` | Field comparisons (after values) |
| `immutable` | Field unchanged after IX |
| `non_negative` | Signed field >= 0 |
| `max_decrease` / `max_increase` | Percentage-bounded change |
| `delta_bound` | Absolute change limit |
| `conserve` | Sum unchanged across accounts |
| `supply_conservation` | Mint supply == sum of balances |
| `lamport_conservation` | SOL sum unchanged |
| `payout_bounded` | Outflow <= formula(state) |
| `aggregate_gte` | Cross-account aggregate check |
| `account_guard` | No unauthorized account create/close |
| `custom` | Developer function with declared CU budget |
| `tx_cumulative_decrease` | Cross-IX drain limit (always runs, even during bypass) |
| `when` | Conditional invariant |
| `monotonic` | Field only increases or only decreases |

## Error Handling Philosophy

Every error path is fail-closed. The pipeline never allows a transaction to proceed in a degraded state:

- Pre-business-logic errors (steps 1-6): transaction reverts, Solana rollback restores all state including the reentrancy guard.
- Post-business-logic errors (steps 8-9): transaction reverts, all mutations rolled back.
- PDA write-back errors (step 10): caught by closure, logged as warnings, execution continues to guard cleanup. Business logic already succeeded — failing to persist a deferred pause flag is less severe than bricking the reentrancy guard.
- Guard cleanup (step 11): always runs on the success path. On error paths, Solana tx atomicity handles rollback.
